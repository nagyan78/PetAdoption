### **技术点 1: MyBatis ORM 映射机制**

#### 代码位置
[[UserMapper.xml](file://d:\MyProject\项目源码\target\classes\mappers\UserMapper.xml)](file://d:\MyProject\项目源码\src\main\resources\mappers\UserMapper.xml#L1-L50)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" 
"http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.xiangmu.mappers.UserMapper">
    
    <!-- 查询所有用户 -->
    <select id="select" resultType="com.xiangmu.pojo.User">
        SELECT * FROM t_user 
        WHERE del='no'
        <if test="username != null and username != ''">
            AND username LIKE CONCAT('%', #{username}, '%')
        </if>
        <if test="xingming != null and xingming != ''">
            AND xingming LIKE CONCAT('%', #{xingming}, '%')
        </if>
    </select>
    
    <!-- 插入用户 -->
    <insert id="insert" useGeneratedKeys="true" keyProperty="uid">
        INSERT INTO t_user(
            username, password, xingming, dianhua, shijian, del, xinyufen
        ) VALUES (
            #{username}, #{password}, #{xingming}, #{dianhua}, 
            #{shijian}, 'no', 60.0
        )
    </insert>
</mapper>
```

#### 技术解析

**1. MyBatis 工作原理**

```
Java 对象 ←→ MyBatis 映射器 ←→ SQL 语句 ←→ 数据库
   ↓           ↓              ↓         ↓
 User     UserMapper    UserMapper.xml  MySQL
```

**2. 命名空间绑定机制**

```java
// Mapper 接口定义
@Mapper
public interface UserMapper {
    public List select(String username, String xingming);
}
```

```xml
<!-- XML 映射文件 -->
<mapper namespace="com.xiangmu.mappers.UserMapper">
    <select id="select" ...>
        <!-- SQL 语句 -->
    </select>
</mapper>
```

**面试标准答案**:
> "MyBatis 通过 `namespace` 将接口与 XML 绑定。接口方法名必须与 XML 中的 [id](file://d:\MyProject\项目源码\src\main\java\com\xiangmu\pojo\Tiezi.java#L10-L10) 一致，这样 MyBatis 就能在运行时动态生成代理对象，调用对应的 SQL 语句。"

**3. 动态 SQL 的强大功能**

```xml
<if test="username != null and username != ''">
    AND username LIKE CONCAT('%', #{username}, '%')
</if>
```

**作用**:
- 防止空指针异常
- 实现条件查询的灵活性
- 避免 SQL 注入风险

**对比硬编码 SQL**:
```sql
-- 硬编码：不灵活
WHERE username = '张三'

-- 动态 SQL：根据参数自动拼接
WHERE 1=1
  AND username LIKE '%张三%'  -- 如果传入了 username
  AND xingming LIKE '%李四%'  -- 如果传入了 xingming
```

**4. 主键回填机制**

```xml
<insert useGeneratedKeys="true" keyProperty="uid">
    INSERT INTO t_user(...) VALUES (...)
</insert>
```

**原理**:
```java
// 调用前 user.getUid() 为 null
User user = new User();
userMapper.insert(user);
// 调用后 user.getUid() 自动获得数据库生成的自增 ID
System.out.println("新生成的用户ID: " + user.getUid());
```

**底层实现**: JDBC 的 `getGeneratedKeys()` 方法

### **技术点 2: 数据库表设计规范**

#### 数据库表结构
```sql
-- 用户表设计
CREATE TABLE t_user (
    uid INT PRIMARY KEY AUTO_INCREMENT,      -- 主键，自增
    username VARCHAR(50) NOT NULL,           -- 用户名，非空
    password VARCHAR(100) NOT NULL,          -- 密码
    xingming VARCHAR(50),                    -- 姓名
    dianhua VARCHAR(20),                     -- 电话
    shijian DATETIME,                        -- 注册时间
    del CHAR(2) DEFAULT 'no',                -- 逻辑删除标记
    xinyufen DOUBLE(10,2) DEFAULT 60.00,    -- 信誉分
    INDEX idx_username (username),           -- 索引优化查询
    INDEX idx_del (del)                      -- 索引加速过滤
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 技术解析

**1. 为什么使用逻辑删除而不是物理删除？**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 物理删除 | 节省空间 | 无法恢复，丢失历史数据 | 临时表 |
| **逻辑删除** | **可恢复，保留历史** | **占用空间** | **业务表 ✅** |

**实际应用场景**:
```java
// 删除用户时，只是标记为已删除
user.setDel("yes");
userMapper.updateXinxi(user);

// 查询时自动过滤已删除的数据
WHERE del = 'no'
```

**2. 索引优化策略**

```sql
-- 单列索引
INDEX idx_username (username)

-- 组合索引 (如果有多个字段经常一起查询)
INDEX idx_query (username, xingming, del)
```

**最左匹配原则**:
```java
// ✅ 能用到索引
WHERE username = '张三'
WHERE username = '张三' AND xingming = '李四'

// ❌ 不能用索引
WHERE xingming = '李四'  // 跳过了 username
```

**3. 字符集选择 utf8mb4**

| 字符集 | 字节数 | 支持 emoji | 推荐度 |
|--------|--------|-----------|--------|
| utf8 | 3 字节 | ❌ | 不推荐 |
| **utf8mb4** | **4 字节** | **✅** | **推荐 ✅** |

**原因**: 现代社交应用需要支持 emoji 表情

### **技术点 3: 关联查询优化**

#### 代码位置
[[DongwuMapper.xml](file://d:\MyProject\项目源码\target\classes\mappers\DongwuMapper.xml)](file://d:\MyProject\项目源码\src\main\resources\mappers\DongwuMapper.xml#L15-L35)

#### 技术解析

**1. INNER JOIN vs LEFT JOIN**

| JOIN 类型 | 行为 | 适用场景 |
|-----------|------|---------|
| INNER JOIN | 只返回匹配的行 | 必须有对应类别 |
| LEFT JOIN | 返回左表所有行 | 允许类别为空 |

**性能对比**:
```sql
-- INNER JOIN (快)
SELECT * FROM t_dongwu d 
INNER JOIN t_leibie l ON d.leibieId = l.id
-- 执行计划：使用索引，扫描行数少

-- LEFT JOIN (慢)
SELECT * FROM t_dongwu d 
LEFT JOIN t_leibie l ON d.leibieId = l.id
-- 执行计划：全表扫描，扫描行数多
```

**2. 别名优化的重要性**

```xml
<l.mingcheng AS leibieName>
```

**好处**:
- 避免字段名冲突
- 提高可读性
- 方便前端使用

```java
// 不使用别名
dongwu.getLeibieMingcheng();  // 冗长

// 使用别名
dongwu.getLeibieName();  // 简洁
```

**3. 排序优化**

```xml
ORDER BY d.fabushi DESC
```

**为什么按时间倒序？**
- 用户更关心最新发布的动物
- 符合产品思维
- 提升用户体验

**性能影响**:
```sql
-- 添加索引优化排序
INDEX idx_fabushi (fabushi)

-- EXPLAIN 分析
EXPLAIN SELECT ... ORDER BY fabushi DESC
-- type: index, key: idx_fabushi
```

### **技术点 4: 事务管理**

#### 代码位置
[[UserService.java](file://d:\MyProject\项目源码\src\main\java\com\xiangmu\service\UserService.java)](file://d:\MyProject\项目源码\src\main\java\com\xiangmu\service\UserService.java#L20-L45)

```java
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;
    
    public void userReg(User user) {
        String username = user.getUsername().trim();
        
        // 检查用户名是否被占用
        List userList = userMapper.findByUsername(username.trim());
        if (userList.size() > 0) {
            throw new ZhanghaoZhanyongException(); // 抛出异常
        }
        
        // 设置默认值
        user.setFujian("");
        user.setShijian(new SimpleDateFormat("yyyy-MM-dd HH:mm")
                        .format(new Date()));
        user.setDel("no");
        
        // 保存到数据库
        int rows = userMapper.insert(user);
    }
}
```

#### 技术解析

**1. Spring 声明式事务**

```java
@Service  // 自动开启事务
public class UserService {
    @Transactional  // 显式声明事务
    public void userReg(User user) {
        // 业务逻辑
    }
}
```

**事务四大特性 (ACID)**:
- **原子性 (Atomicity)**: 要么全部成功，要么全部失败
- **一致性 (Consistency)**: 数据从一个一致状态到另一个一致状态
- **隔离性 (Isolation)**: 并发事务互不干扰
- **持久性 (Durability)**: 事务提交后永久保存

**2. 事务传播行为**

```java
@Transactional(propagation = Propagation.REQUIRED)  // 默认
public void methodA() {
    // 如果已有事务则加入，否则新建事务
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void methodB() {
    // 总是新建事务，挂起现有事务
}
```

**实际应用场景**:
```java
// 订单创建流程
@Transactional
public void createOrder() {
    // 1. 扣减库存 (子事务)
    decreaseStock();
    
    // 2. 生成订单 (主事务)
    saveOrder();
    
    // 3. 增加销量 (子事务)
    increaseSales();
}
```

**3. 异常回滚机制**

```java
try {
    userService.userReg(user);
} catch (ZhanghaoZhanyongException e) {
    // 自动回滚，数据库不会插入数据
    System.out.println("注册失败：" + e.getMessage());
}
```

**Spring 回滚规则**:
- ✅ RuntimeException: 自动回滚
- ✅ Error: 自动回滚
- ❌ Checked Exception: 默认不回滚 (需配置 `rollbackFor`)

### **技术点 5: 连接池优化**

#### 配置文件
[[application.properties](file://d:\MyProject\项目源码\src\main\resources\application.properties)](file://d:\MyProject\项目源码\src\main\resources\application.properties#L12-L17)

```properties
# Druid 连接池配置
spring.datasource.type=com.alibaba.druid.pool.DruidDataSource
spring.datasource.druid.initial-size=5
spring.datasource.druid.min-idle=5
spring.datasource.druid.max-active=20
spring.datasource.druid.max-wait=60000
spring.datasource.druid.validation-query=SELECT 1
spring.datasource.druid.test-while-idle=true
```

#### 技术解析

**1. 为什么需要连接池？**

**无连接池的问题**:
```
请求 1 → 创建连接 (耗时 100ms) → 执行 SQL → 销毁连接
请求 2 → 创建连接 (耗时 100ms) → 执行 SQL → 销毁连接
...
总耗时 = N × 100ms
```

**有连接池的优势**:
```
启动时 → 预创建 5 个连接 (耗时 500ms)
请求 1 → 获取连接 (耗时 1ms) → 执行 SQL → 归还连接
请求 2 → 获取连接 (耗时 1ms) → 执行 SQL → 归还连接
...
总耗时 = 500ms + N × 1ms
```

**性能提升**: 100 倍！

**2. Druid 参数详解**

| 参数 | 含义 | 推荐值 | 作用 |
|------|------|--------|------|
| `initial-size` | 初始连接数 | 5 | 启动时创建的连接数 |
| `min-idle` | 最小空闲连接 | 5 | 保持的最小连接数 |
| `max-active` | 最大活跃连接 | 20 | 同时可用的最大连接 |
| `max-wait` | 最大等待时间 | 60s | 获取连接的超时时间 |

**3. 连接检测机制**

```properties
spring.datasource.druid.validation-query=SELECT 1
spring.datasource.druid.test-while-idle=true
```

**工作原理**:
```
定时线程 → 每 60 秒执行一次 → SELECT 1
    ↓
验证连接是否有效
    ↓
无效连接 → 关闭并重建
```

**好处**:
- 防止连接失效
- 保证连接可用性
- 不影响业务性能

### **技术点 6: 数据库设计范式**

#### 实际案例

**第一范式 (1NF) - 字段不可再分**
```sql
-- ❌ 不符合 1NF
CREATE TABLE t_user (
    contact_info VARCHAR(200)  -- 包含电话、地址、邮箱
)

-- ✅ 符合 1NF
CREATE TABLE t_user (
    dianhua VARCHAR(20),
    dizhi VARCHAR(100),
    youxiang VARCHAR(50)
)
```

**第二范式 (2NF) - 消除部分依赖**
```sql
-- ❌ 不符合 2NF
CREATE TABLE t_order (
    order_id INT,
    user_id INT,
    username VARCHAR(50),  -- 依赖 user_id，而非 order_id
    PRIMARY KEY (order_id, user_id)
)

-- ✅ 符合 2NF
CREATE TABLE t_order (
    order_id INT PRIMARY KEY,
    user_id INT
)

CREATE TABLE t_user (
    user_id INT PRIMARY KEY,
    username VARCHAR(50)
)
```

**第三范式 (3NF) - 消除传递依赖**
```sql
-- ❌ 不符合 3NF
CREATE TABLE t_user (
    uid INT PRIMARY KEY,
    department_id INT,
    department_name VARCHAR(50),  -- 传递依赖
    department_location VARCHAR(100)  -- 传递依赖
)

-- ✅ 符合 3NF
CREATE TABLE t_user (
    uid INT PRIMARY KEY,
    department_id INT
)

CREATE TABLE t_department (
    department_id INT PRIMARY KEY,
    department_name VARCHAR(50),
    department_location VARCHAR(100)
)
```
